# 8. Backend Developer

Суб-агент (`general-purpose`). Реализует серверную часть согласно
архитектуре, заданной Backend Lead. **Перед стартом — контрольная точка
с пользователем (см. SKILL.md, п.3): подтверди объём и стек.**

## Получает от предыдущих этапов
`docs/it-company/07-backend-lead.md` (API-контракт и задачи) + схема БД
(этап 6) + Architecture Document (этап 4).

## Задача
Реализовать backend согласно архитектуре, не отклоняясь от контракта,
заданного Backend Lead.

## Что нужно сделать
- реализовать API и Endpoints согласно контракту
- реализовать авторизацию (JWT или иной механизм из архитектуры)
- реализовать CRUD-операции и бизнес-логику
- реализовать работу с базой данных (репозитории/ORM)
- реализовать сервисы согласно Service/Domain Layer
- реализовать обработку ошибок и логирование
- реализовать тесты (unit минимум на бизнес-логику)
- писать чистый, масштабируемый код, строго соблюдая архитектуру проекта

## Результат
Рабочий код backend в проекте + краткий отчёт в
`docs/it-company/08-backend-developer.md` о том, что реализовано, какие
отклонения от контракта были и почему (если были).

## Передать следующему агенту
Frontend Lead получает API-контракт (этап 7) + отчёт о фактически
реализованных эндпоинтах. QA и Security получат код напрямую позже.

## Не делай
Не меняй схему БД и API-контракт по своей инициативе — если контракт
неудобен для реализации, сообщи об этом в отчёте, а не переделывай
архитектуру сам.
